# vike-i18n

The **localization core** for vike-data: a framework-agnostic layer on top of Vike's
locale routing that lets extensions ship their own translated strings and compose them
into the app the same way `schemas` and `themes` compose. Zero framework imports; the
React binding (`vike-i18n/react`) is a thin wrapper.

Translations resolve in three tiers, all behind one lookup:

1. **Inline English** — every extension ships its English strings inline, so the UI
   renders standalone with no i18n runtime. English is the universal fallback.
2. **Bundled packs (tier 1, #76)** — common languages ship as subpaths
   (`vike-auth/fr`, `vike-auth/ar`) and light up via zero-config
   [`locales: [...]`](#zero-config-locales). Unused locales tree-shake out.
3. **Committed translations (tier 2, #102)** — the **long tail** of languages is
   AI-generated by [`vike translate`](#vike-translate) into a single
   `translation.json` you commit. This keeps `node_modules` small (extensions don't
   ship every language) while still reaching any locale.

Precedence at lookup time: **committed `translation.json` → bundled pack → inline
English**. A committed translation overrides a bundled pack for the same key; a missing
key falls back down the chain.

## `vike translate`

`vike translate` is the tier-2 developer command — a **codegen-style gate**, the i18n
twin of `vike-schema`'s `gen` / `gen:check`. It is deterministic and re-runnable, and
no AI runs at build, CI, or runtime: translation is a developer action you re-run when
source texts change.

```bash
# Translate every installed extension's texts (+ your app's) into fr and ar.
npx vike-translate --locales en,fr,ar

# CI drift gate — fails if translation.json is missing or stale entries. Never writes.
npx vike-translate --check
```

It writes/updates `translation.json` in your repo root (commit it):

```jsonc
{
  // staleness ledger: the English-source hash each translation was generated from
  "$sources": { "auth.signIn": "1a2b3c4d", "auth.email": "5e6f7a8b" },
  "ar": { "auth.signIn": "تسجيل الدخول إلى {app}", "auth.email": "البريد الإلكتروني" },
  "fr": { "auth.signIn": "Connexion à {app}", "auth.email": "E-mail" }
}
```

Why committed (not runtime AI): it's **editable** (fix a bad machine translation by
hand — `$sources` doesn't change, so your edit survives re-runs), **reviewable** (shows
up in PRs), and **deterministic**.

### How it finds source texts: `exports["texts"]`

An extension advertises its English source strings as plain JSON via package.json:

```jsonc
// packages/vike-auth/package.json — "./texts" sits alongside the package's other
// subpaths (Node forbids mixing a bare "texts" key with "." subpath keys).
{ "exports": { ".": "./index.js", "./texts": "./texts.json" } }
```

```jsonc
// packages/vike-auth/texts.json — the source { key: english } catalog
{ "auth.signIn": "Sign in to {app}", "auth.email": "Email" }
```

Because it's JSON, `vike translate` enumerates every installed extension's texts by
**reading** package.json + the JSON — never importing (executing) extension code, never
booting a bundler. Your app can advertise its own `texts` too (its UI strings); the
app's catalog is merged last, so it can override any extension's source string.

### Staleness & incrementality

Each run only (re)translates what changed:

- **Missing** — a locale has no string for a key yet.
- **Stale** — the English source changed since the committed translation was generated
  (detected by hashing the source and comparing to `$sources`). Only the changed key is
  re-translated, in every locale.
- **Removed** — a key gone from the source is pruned from the file.

`--check` reports a non-zero exit when any of the above is true, so a stale
`translation.json` fails CI exactly like `gen:check`.

### Provider

The default provider translates via Claude (`@anthropic-ai/sdk`, an **optional** peer
dep) using `ANTHROPIC_API_KEY`. It's only needed in write mode — `--check` and
English-only apps never touch it. Swap it for your own by passing a `translate`
function to `runTranslate` (`vike-i18n/cli`):

```js
import { runTranslate } from 'vike-i18n/cli'

await runTranslate({
  locales: ['en', 'fr', 'de'],
  // (job) => { [key]: translatedString }; job = { locale, items: [{ key, source }] }
  translate: async ({ locale, items }) => myProvider(locale, items),
})
```

## Example: a non-bundled locale

`vike-auth` bundles `fr` and `ar` as subpaths (tier 1). To reach a locale it does *not*
bundle — say German — you don't need a new package; just translate:

```bash
npx vike-translate --locales en,de
```

This reads `vike-auth`'s `texts.json`, AI-translates `auth.*` into German, and commits
`translation.json` with a `de` block. At runtime the vike-i18n Vite plugin folds that
`de` block into the locale dictionary (after any bundled packs, so it wins), and `de`
appears in the locale picker — no extension change required.

## Zero-config `locales`

The app declares its languages once; every installed extension's matching tier-1 pack
auto-includes, and any committed tier-2 locale in `translation.json` is folded in too:

```js
// app config
locales: ['en', 'fr', 'ar', 'de']
```

Add `vikeI18n()` from `vike-i18n/plugin` to your Vite config so the virtual packs
module (which inlines the committed translations) resolves.
