vike-notifications: bind the in-app feed to a non-default auth subject with `notificationsGuard` (the downstream "which subject" seam, #279 / #207 P3) — the notifications half of the same seam vike-storage's `storageGuard` opened. An app that runs vike-auth's named guards (#267) can now own notifications by a guard other than the default user: set `notificationsGuard: '<name>'` and the `notifications.user_id` FK target follows that guard's subject table (e.g. `clients` instead of `users`), while the `/notifications` feed endpoint resolves the reader from that guard's session cookie and a bare-id `notify()` hydrates the notifiable from that guard's subject (both via the new `VIKE_NOTIFICATIONS_GUARD` runtime knob). The schema is now a computed contribution parameterized by the resolved config, the same shape vike-stripe's `subscriptionSchemas({ segment })` and vike-storage's `uploadsSchemas` use; the FK column stays `user_id`, only its target moves. Unset = the default `users` subject, byte-for-byte unchanged, so the common single-subject app is untouched. The build-time `notificationsGuard` and the runtime `VIKE_NOTIFICATIONS_GUARD` mirror vike-storage's `storageGuard`/`VIKE_STORAGE_GUARD` split.
