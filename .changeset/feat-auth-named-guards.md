vike-auth: named, multi-instance guards (#267, epic #207 P2). An app can now run vike-auth more than once over different subjects/tables, Laravel-"guards" style: `defineGuard('admin', { table: 'admins' })` declares an independent audience with its own session cookie (`vike_auth_session__admin`), endpoint namespace (`/admin-auth/*`), tables (`admins` + the `<name>_sessions` / `<name>_login_tokens` defaults, each overridable via `sessionTable` / `loginTokenTable`), and resolved user (`pageContext.guards.admin.user`). Guards never share a store or session, so two audiences can be signed in at once in one browser with no cross-talk, and logout on one leaves the other untouched.

The named-guards tier is opt-in and purely additive: install `vike-auth/react/guards` (a dispatcher middleware + a render hook + the `authGuard` page meta) alongside `vike-auth/react`, declare guards with `defineGuard`, spread `guard.schemas` into the cumulative `schemas` point, and own the per-guard login routes (pointing each at the extension's `GuardLoginPage`). An app that declares no guards keeps the byte-for-byte single-subject default (`/login`, `/auth/*`, `vike_auth_session`, `pageContext.user`). Each guard's tables merge + derive across all three ORMs alongside the default's. The downstream "which subject" seam for push/notifications/storage (P3) and org-owned arity (#250) remain later phases.
